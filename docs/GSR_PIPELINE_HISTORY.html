The Pattern the Archaeology Revealed

The archaeology.py tool processed all 879 conversations and found a single dominant macro-pattern: automation planning cycles that produced design, architecture, and partial code — but no shipped production systems. This was not occasional drift. It was the modal outcome of a planning session.

🔄
The Design → Scaffold → Never Ship Loop
Post-production system (May 2026). Apps Script email automation (April 2026). Guest Portal (Feb 2025). Each started with a full design session. None confirmed in steady production use at close of session.
🆕
Latest-Shiny-Tool Syndrome
n8n → Notion → Supabase → n8n.cloud all evaluated within one week. Each new tool required starting the design from scratch. Context switched faster than code shipped.
📋
Planning Without a Named Deliverable
Extended design sessions produced documentation and scaffolding code — but no named file, feature, or date that would mark the session as done. Planning felt like progress.
🔁
400+ Recurring Friction Points Never Fixed at Source
Wrong hook structure, wrong character counts, wrong credentials — each required session-by-session correction rather than one structural fix baked into the system.

The Rules That Came From It

The Anti-Churn Rule (CLAUDE.md)

Before designing, scoping, or building any automation in a session — name the deliverable and the ship date out loud first. If you can't answer both, the session is planning only. Planning sessions end with a named task in docs/AUTOMATION_ROADMAP.md, not code in the repo.

The One-Feature Strategy (ADR-0012)

The lower-thirds approval workflow ships before any other feature is touched. This rule is structural — written into the architecture decision record. It's also why the dashboard exists at all: it's the first thing to break the pattern.

07
How an Episode Moves Through the System
Full pipeline from "guest booked" to "lower thirds in ProPresenter"
1
Guest Booked
Daniel uses Prompt 01 from /toolkit (live roster injection prevents suggesting recently-aired guests). Outreach email sent. Guest confirmed and assigned to interview_1 or interview_2 in the episode_guests table.
2
Script Written in Google Drive
Full episode script written in a Drive folder named "S3 EP12 - Guest Name Interview." Lower-thirds copy written directly in the script doc in ALL CAPS. Rundown built in Rundown Creator (most-used tool at 1,750+ sessions).
3
Extraction Prompt Runs in Claude.ai
Daniel opens Claude.ai with the Google Drive connector enabled. Clicks "Copy extraction prompt" in /import, which pre-builds a prompt. That prompt reads all docs in the episode folder and outputs structured JSON: episodes[], graphics[], rejected[].
4
/import — Dry Run Then Live
JSON pasted into the import form. Dry-run first: shows episode count, graphic count, conflicts, rejected items. Explicit confirmation required. Live import upserts episodes, inserts graphics with status=pending_review, stores variation_1 with source doc as provenance. ← Current Stage 7 blocker: JSON field names misaligned with DB schema.
5
/lower-thirds — Review Grid
Every graphic lands with status=pending_review. Each card shows: text, character counter (green 55–65 / amber under 55 / red over 65), Approve, Reject, and Regenerate. Regenerate calls the Claude API → returns 3 variation options, all stored to graphics_variations history. Daniel picks one and approves.
6
/approved — ProPresenter Queue
Approved graphics grouped by episode → segment → beat_number. Each row has a Copy button (copies the ALL CAPS approved text as plain string — exactly what gets pasted into ProPresenter) and an "In ProPresenter" toggle that flips atomically via SQL RPC.
7
ProPresenter Loaded — TD Takes Over
Daniel manually pastes each line into ProPresenter slides and ticks the checkbox. On shoot day, the TD calls them live. Phase 2 target: automate this handoff via ProPresenter API — currently blocked by "The David Rule" (must run on test machine first, never touch production without David's explicit sign-off).
Where Claude API Fits

Claude API (claude-opus-4-7) is used for exactly one step: regenerating alternative lower-third text at step 5. The extraction prompt that converts Drive docs to JSON runs in Claude.ai browser with Drive connector — it is NOT an API call from the app. Everything else is plain Next.js + Supabase.

The Lower Thirds Disconnect
What was originally intended vs. what was built — and why it matters
Original Intent (Spec)
Image Review Workflow
FEATURE_1_LOWER_THIRDS.md
  • Jakob uploads finished PNG from Photoshop
  • Dashboard shows thumbnails for visual approval
  • Daniel (or David) approves the actual graphic file
  • Approved PNGs queue for ProPresenter load
  • Text copy is metadata attached to the graphic
The graphic file is the artifact being approved.
What Was Built
Text Review Workflow
Built May 23–27, 2026
  • Daniel extracts LT copy from script JSON
  • Dashboard shows text + character count
  • Daniel approves/rejects/regenerates the TEXT
  • Approved text copies to clipboard for ProPresenter
  • No PNG images — __text_only__ sentinel
The text content is the artifact being approved.

Why It Changed

The text-only path was added as Stage 6.5 — a bootstrap so the dashboard could run on real content immediately before Jakob changed his PNG workflow. The first two real episodes had no completed graphics yet. Rather than block, Daniel used the dashboard for what was immediately useful: approving and refining the copy.

The graphic-approval vision isn't gone — it's Phase 2 of Feature 1, when ProPresenter API push is implemented. But the primary use case has drifted from the spec, and that's worth naming explicitly.

Technical Debt Created by This Drift
  • current_image_url is NOT NULL with a __text_only__ sentinel — documented hack, scheduled for cleanup post-Stage 7
  • /upload PNG page is still live but marked legacy — to be retired after 2 text-only episodes
  • FontEditor component (font family/size/color per graphic) was added to /approved — not in original spec, and not relevant until real PNGs are in the system
  • BUILD_STATUS.html is stale by 14 migration entries

What Was Learned
The distilled lessons from 879 conversations, three pivots, and one security incident
01
Name the deliverable before building
If you can't say what file or feature will exist at the end — and when it goes live — the session is planning. Planning ends with a task in the roadmap, not scaffolding in the repo.
02
One working feature beats a perfect architecture
Three ADRs before one production feature shipped. The constraint "one feature first" is what finally broke the pattern. The constraint is the feature.
03
Security is not optional
One chat session with credentials exposed triggered an IT incident, rotated 40+ devices, and forced a complete architecture pivot. The CLAUDE.md Security Rules exist because of this exact event.
04
Specificity is what makes AI output usable
400+ recurring friction points all traced to AI working without specific context. Every solved problem was solved by baking the constraint into the system — not correcting it session by session.
05
Build for the workflow, not the tool
n8n was chosen before the workflow was defined. Notion before the data model was clear. Both abandoned because the workflow had requirements the tool couldn't meet. The current stack was chosen because Feature 1 required exactly what Supabase provides.
06
Pragmatic drift can be productive
The text-only path was an unplanned workaround that kept the project moving. It was scope drift — but it shipped something useful before the "intended" flow was possible. Document the drift, schedule the cleanup, keep moving.
Where the Project Stands Right Now

The GSR pipeline is one episode run away from Feature 1 being complete. The import JSON schema mismatch is the only technical blocker for Stage 7. Once fixed and one episode runs through import → review → approve → ProPresenter copy, Feature 1 is done. The schema for the next three features (guests/booking, distribution, social) is already in the database waiting for UI. The system that burned down 879 times now has 42 migrations, 9 routes, a live Vercel deployment, and a draft PR with CI running.
