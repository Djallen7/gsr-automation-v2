Agent Teams let you coordinate multiple
Claude Code instances working together.
One session acts as the team lead.
It creates teammates, assigns tasks, and
synthesizes results.
Each teammate is a full, independent
Claude Code session with its own context
window.
They work in parallel and here's the key
difference.
They can talk directly to each other.
Now, you might be thinking
don't sub-agents already do this?
Not quite.
Sub-agents are workers that report back
to one boss.
They can never talk to each other.
Agent Teams are fundamentally different.
Teammates share a task list claim work
independently, and send messages
directly between themselves.
Think of sub-agents as contractors
reporting to a manager.
Agent teams are a self-organizing squad.
The architecture has four key
components.
The team lead, Claude your main Claude
session that orchestrates everything.
Teammates
separate Claude instances
that each own their tasks.
A shared task list
where work items get created, claimed,
and completed.
And a mailbox system
for direct communication
between any agents.
Teams and tasks are stored locally in
your .claude directory.
Agent teams are experimental and
disabled by default.

Open your settings
.json
and add

agent teams set to one in your env
block.
Or set it as an environment variable in
your shell.
That's literally all you need.

The shared task list is the coordination
backbone.
The lead creates tasks and teammates
work through them.
Tasks have three states:
pending, in progress, and completed.
Tasks can also depend on other tasks.
A pending task with unresolved
dependencies
can't be claimed until those
dependencies are done.
The system
handles dependency resolution
Two ways tasks get assigned.
The lead explicitly assigns tasks to
specific teammates,
or
teammates self-claim.
After finishing one task,
they grab the next
unassigned
unblocked task on their own.
Task claiming
uses file locking
to prevent race conditions
when multiple teammates try to claim the
same task simultaneously.
No conflicts, no double work.
Parallel code review.
Create a team with three reviewers.
One focused on security implications,
one checking performance impact,
one validating test coverage.
Each reviewer works from the same PR,
but applies a different filter.
A single reviewer gravitates toward one
type of issue.
With three specialized agents, security,
performance, and test coverage all get
thorough attention simultaneously.
Competing hypotheses debugging.
Users report the app exits after one
message.
Spawn five agent teammates to
investigate different hypotheses.
Have them talk to each other to try to
disprove each other's theories.
Like a scientific debate.
Sequential debugging suffers from
anchoring.
Once one theory is explored, you're
biased.
Multiple investigators actively
disproving each other means the
surviving theory is much more likely to
be the actual root cause.
Cross-layer coordination.
When you need changes spanning front
end, back end, and database,
each owned by a different teammate,
they work simultaneously in their own
domains,
communicate through the shared task
list,
and the lead ensures
everything integrates cleanly.
What used to take days of sequential
work
now happens in parallel in minutes. Team
sizing matters.
Start with three to five teammates for
most workflows.
That balances parallel work with
manageable coordination.
Having five to six tasks per teammate
keeps everyone productive without
excessive context switching.
If you have 15 independent tasks,
three teammates is your sweet spot.
Three focused teammates almost always
outperform five scattered ones.
Token reality check.
Each teammate has its own context
Token usage scales linearly with team
size.
For research, review, and new feature
work,
the extra tokens are usually worth it.
For routine tasks, a single session is
more cost-effective.
Size tasks appropriately.
Too small and coordination overhead
kills you.
Too large and teammates work too long
without check-ins.
Find that sweet spot of self-contained
units that produce a clear deliverable.
Use hooks for quality gates.
Teammate idle fires.
When a teammate is about to go idle,
exit with code two to send feedback
and keep them working.
Task completed fires.
When a task is being marked complete,
exit with code two to prevent completion
and send feedback.
This means you can enforce rules
Require tests before marking done.
Validate code style.
Or
check that documentation was updated.
Five power tips.
One.
Always use Claude.md files.
Teammates load them automatically for
project-specific guidance.
Two.
Pre-approve common permissions
before spawning teammates to avoid
constant approval prompts.
Three.
Avoid two teammates editing the same
file.
Break work so each owns different files.
Four.
If the lead starts implementing instead
of delegating,
tell it to wait for teammates.
Five. Start with research tasks before
moving to parallel implementation.
