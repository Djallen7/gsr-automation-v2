## STANDARDIZATION RULES (the going-forward norm)

These are the standards applied to all 16 rows. Use them for every future episode row too.

1. **Guest full name (display):** `First Last`, with an optional single leading credential token taken verbatim from the `guests.title` field when the table sets one consistently (`Dr.`, `Col.`). Format: `Dr. First Last`. Do not invent or add credentials the table leaves blank. No trailing post-nominals in the name field.
2. **`episodes.guest_name` (text field on the episode):** the two interview guests joined with the word "and", each in full display form, Interview 1 guest first: `Dr. First Last and Dr. First Last`. This text field is for human display only.
3. **`episode_guests` (relational links):** the real source of truth for guest-to-episode. ONE row per guest per episode: `episode_id`, `guest_id` (the canonical uuid below), `segment` = `interview_1` or `interview_2`. The `guest_name` text and the relational rows must always agree.
4. **Affiliation:** rendered as `Job Title, Organization` from the `guests` table (e.g. "Director of Research, Institute for Creation Research"). If `organization` is null, use the job title alone. Title Case for organizations; do not abbreviate. This is recorded here for review; it is NOT written to `episodes` (it lives on the `guests` row).
5. **Dates:** ISO `YYYY-MM-DD`. Air dates are the DB/Airing-Schedule values (authoritative). Shoot dates are given only where a single unambiguous calendar date exists; otherwise marked NEEDS DANIEL (see flags), never guessed.
6. **Titles:** kept VERBATIM as published in the DB. Published titles are not reworded. The only changes are confirmed typo fixes, and those are listed as flags for Daniel to apply, not silently changed here. The DB title text is reproduced exactly (including its `&`, quotes, and punctuation).
7. **Capitalization and trimming:** all values trimmed of leading/trailing whitespace; names in Title Case; no double spaces.

## RUNDOWN CREATOR INGEST + WHAT WAS WRITTEN (2026-06-08)

Source: Rundown Creator API (https://www.rundowncreator.com/davidrives/API.php), getRundowns / getRows / getScript. The API works; credentials needed whitespace trimming.

### Key structural finding (READ THIS BEFORE TRUSTING ANY rc_rundown_id FOR EP1-17)

GSR did NOT use one rundown per episode for Season 3 episodes 1-17. The clean one-rundown-per-episode convention with "S03_EpNNN" in the title only begins at **Ep021** (the May block, RundownID 79-84 = Ep021-025, taped 2026-05-28). For ep1-17 the production used:

- **Interview-taping rundowns** that batch MANY episodes' interviews into one taping-day rundown (e.g. RundownID 46 "GSR MARCH 6th INTERVIEWS", taped 2026-03-06, contains the interviews for episodes 8, 9, 11, 12, 13 and a Brian Thomas interview, all in one rundown). These name guests in the row slugs.
- **Monthly assembly shells** "Show 1-5 - February/March" and "April: Show 1-5" (RundownID 32-36, 54-58, 64-68) that hold the full broadcast run-of-show (Show Intro, Opening Monologue, Interview 1, The Heavens Declare, Interview 2, Closing) but DO NOT name the guest in the interview row slugs, so they cannot be matched to a specific episode by slug alone.

Consequence: there is no single, unambiguous `rc_rundown_id` nor a single `shoot_date` for any of ep1-17. Assigning one would be a guess. Per the David Rule, those two fields were therefore **left untouched in the DB** and are recorded below as a research mapping only, for Daniel to confirm.
