# Open-Source Building Blocks for Christian Ministry Post-Production Automation

A production-ready evaluation of 40+ GitHub projects for Genesis Science Report's automated video distribution system running on n8n, prioritizing non-developer accessibility and battle-tested reliability.

## Executive Overview

**The good news**: For 11 of 14 components, mature open-source solutions exist with excellent n8n integration paths. **The challenge**: Rumble automation remains sparse, and specialized services like Fireside.fm require custom browser automation. **The strategy**: Use official SDKs where available (YouTube, Dropbox), proven Node.js libraries for infrastructure (Chokidar, BullMQ), and AI-powered Whisper implementations for transcription—all wrapped in a modern Next.js dashboard with native n8n approval workflows.

This research evaluated projects based on five critical factors: active maintenance (commits in last 6 months), permissive licensing (MIT/Apache preferred), real production usage, JavaScript/Python compatibility with n8n, and integration complexity for non-developers working with AI assistance. The recommended stack balances simplicity with enterprise-grade reliability, avoiding bleeding-edge experiments in favor of boring, proven technology.

## Component 1: File Watching on NAS

**Recommended Project**: Chokidar  
- **URL**: https://github.com/paulmillr/chokidar  
- **Stars**: 11,900 | **Last commit**: Nov 25, 2025 | **License**: MIT  
- **Language**: TypeScript (99.7%)

**Why this one**: Chokidar dominates file watching with 30 million dependent repositories and 13 years of production hardening. It's the file watcher used by webpack, gulp, and other major build tools. For NAS monitoring specifically, the `awaitWriteFinish` option ensures videos are fully written before triggering (preventing partial file uploads), while `usePolling: true` works reliably on network shares where native OS events fail. Version 5 (Nov 2025) brings ESM-only architecture with improved performance.

**How to use**: Library import via npm

**Integration with n8n**: **Easy** - Pure JavaScript module that runs directly in n8n's Node.js environment. Use n8n's Function node to instantiate Chokidar, configure paths to NAS folders (via SMB/NFS mounts), and emit events to n8n workflow triggers. Example: Watch `/mnt/nas/incoming-videos/*.mp4`, trigger workflow on "add" event with file path.

**Code snippets to grab**: Core watcher initialization from `examples/` directory, specifically the `awaitWriteFinish` configuration for large file handling and the `usePolling` fallback for network drives. The atomic write detection logic ensures chunked file transfers complete before triggering.

**Honorable mentions**:  
- **watchdog** (7,300 stars, Python): Excellent if building Python-based automation, but adds language complexity for n8n workflows  
- **fswatch** (5,500 stars, C++): Ultra-lightweight CLI tool, but requires shell scripting and GPL license is more restrictive

**Gotchas**: On Linux systems watching large directories (10,000+ files), you may hit `ENOSPC` errors (file handle exhaustion). Solution: increase system limits with `echo fs.inotify.max_user_watches=524288 | sudo tee -a /etc/sysctl.conf`. For NAS shares, always use `usePolling: true` with `interval: 1000` (1-second checks) to avoid missing events on network filesystems.

## Component 2: YouTube Upload Automation

**Recommended Project**: youtubeuploader  
- **URL**: https://github.com/porjo/youtubeuploader  
- **Stars**: 867 | **Last commit**: Aug 14, 2025 (v1.25.5) | **License**: Apache 2.0  
- **Language**: Go (100%)

**Why this one**: Unlike tokland/youtube-upload (Python-based, GPL license), youtubeuploader compiles to a single binary executable with zero dependencies—perfect for n8n Execute Command nodes. JSON metadata files integrate seamlessly with n8n's data transformation, and resumable chunked uploads handle large sermon videos reliably. The Apache 2.0 license permits commercial ministry use without restrictions, and active maintenance (August 2025 release) ensures compatibility with YouTube API changes.

**How to use**: Run as-is via command-line execution

**Integration with n8n**: **Easy** - Download precompiled binary for your platform (Linux/Mac/Windows), place in accessible path, call via n8n Execute Command node. Pass metadata as JSON file (generated by AI metadata component), video path, and OAuth token. Example: `youtubeuploader -filename /path/video.mp4 -metaJSON metadata.json -cache request.token`

**Code snippets to grab**: Not applicable for binary usage, but review `metadata.json` schema in documentation for structuring AI-generated metadata (title, description, tags, category, privacy status, scheduled publish time).

**Honorable mentions**:  
- **tokland/youtube-upload** (2,200 stars, Python): More popular but GPL license and Python dependency add complexity  
- **googleapis Node.js client** (11,200+ stars): Official Google SDK, best for custom workflows requiring programmatic control, but requires more coding than CLI tools

**Gotchas**: **Critical**: YouTube API restrictions apply regardless of tool. New Google Cloud projects created after July 2020 are limited to "private" uploads until you complete YouTube audit (can take weeks). Default quota allows only ~6 videos per 24 hours. You'll need to create your own OAuth 2.0 application in Google Cloud Console—the default `client_secrets.json` was revoked. Budget 30-60 minutes for initial OAuth setup (one-time, browser required), then fully automated afterward.

**Code snippets to grab**: OAuth token refresh implementation from `examples/javascript/auth/index.html`, chunked upload session handling from `examples/javascript/upload/index.js`, error handling and retry logic for network failures. **Code snippets to grab**: Create custom wrapper combining file upload pattern from Playwright's `input-file.spec.ts` examples, form filling from authentication examples, and wait strategies from network idle examples. **Honorable mentions**:

**Gotchas**: Fireside.fm may update upload UI without notice, breaking automation (monthly maintenance likely required). ## Component 7: StreamHoster Upload (FTP)

**Recommended Project**: basic-ftp  
- **URL**: https://github.com/patrickjuchli/basic-ftp  
- **Stars**: 724 | **Last commit**: April 2026 | **License**: MIT  
- **Language**: TypeScript (58%), JavaScript (41.5%)

**Why this one**: Best modern FTP client for Node.js with promise-based async/await API, FTPS (TLS) support for secure transfers, and zero dependencies beyond Node.js. The `uploadFrom()` method handles large video files efficiently, and `ensureDir()` creates remote directory structures automatically. Resume support recovers from failed uploads. While ssh2-sftp-client (~1,000 stars) is more secure, basic-ftp's simplicity and active maintenance (April 2026) make it ideal for straightforward FTP workflows. TypeScript definitions provide excellent autocomplete for AI-assisted coding.

**How to use**: Library import via npm

**Integration with n8n**: **Easy** - Install via npm, use n8n Code node to implement FTP upload. Simple three-step workflow: `client.access()` with credentials, `client.uploadFrom(localPath, remotePath)`, `client.close()`. The promise-based API works naturally with async/await in n8n's JavaScript executor. FTPS encryption (`secure: true`) protects credentials and content in transit.

**Code snippets to grab**: Connection setup with FTPS from README examples, error handling and retry logic from issues discussions, directory creation patterns using `ensureDir()`, progress tracking via event listeners. The timeout configuration is critical for large video files: `client.timeout = 300000` (5 minutes prevents premature disconnections).

**Honorable mentions**:  
- **ssh2-sftp-client** (~1,000 stars, MIT): Use if StreamHoster supports SFTP (more secure than FTP/FTPS)  
- **n8n's built-in FTP node**: May handle basic uploads without custom code

**Gotchas**: FTP is an ancient protocol with known security limitations—always use FTPS (`secure: true`) when available, never plain FTP for sensitive content. Active Mode FTP doesn't work (not supported by basic-ftp); ensure StreamHoster uses Passive Mode. Some hosting providers impose connection limits (typically 5-10 concurrent connections); implement connection pooling or rate limiting if uploading many files. Validate file integrity post-upload with checksums—FTP has no built-in verification.

## Component 8: Audio/Video Transcription

**Recommended Project**: faster-whisper  
- **URL**: https://github.com/SYSTRAN/faster-whisper  
- **Stars**: 22,900+ | **Last commit**: Active (2024-2025) | **License**: MIT  
- **Language**: Python

**Why this one**: Delivers 4x speed improvement over OpenAI's original Whisper with identical accuracy by using CTranslate2 inference engine. **Critical advantage**: Handles video files natively (extracts audio via PyAV without FFmpeg dependency). Batch processing mode achieves 17-second transcription for 13-minute videos (GPU). The large-v2 model balances accuracy and speed for broadcast content. While whisper.cpp (49,500 stars) is faster on CPU, faster-whisper's Python ecosystem and straightforward API make it easier to wrap in HTTP service for n8n integration. WhisperX (5,000+ stars) adds speaker diarization but increases complexity—save that for interviews requiring speaker labels.

**How to use**: Library import in Python service

**Integration with n8n**: **Medium** - Deploy faster-whisper in Docker container with Flask/FastAPI wrapper, expose HTTP endpoint. n8n calls endpoint with video file path/URL, receives transcript JSON. Workflow: n8n HTTP Request node → POST video path → faster-whisper processes → returns transcript with timestamps. Use `batch_size=8` on GPU for optimal speed, `int8` quantization if GPU memory limited.

**Code snippets to grab**: Model initialization with quantization settings from README, batch transcription loop from examples, VAD filtering configuration (removes silence), word-level timestamp generation. The `awaitWriteFinish` option ensures video files are completely written before transcription starts (pairs with Chokidar file watcher).

**Honorable mentions**:  
- **whisper.cpp** (49,500 stars, MIT, C++): Faster on CPU, excellent for resource-constrained environments, but requires compilation and has steeper integration curve  
- **WhisperX** (BSD-4-Clause): Add later if speaker diarization needed (identifies "Pastor", "Guest", etc.)

**Gotchas**: Requires Python 3.9+ and GPU for real-time performance (1-hour video = 3-10 minutes on GPU vs. 30-60 minutes on CPU). CUDA drivers needed for NVIDIA GPU acceleration (budget 30 minutes for setup). For large-v2 model, GPU needs ~4.5GB VRAM (most modern GPUs sufficient). Transcription quality degrades with background music/noise—consider audio preprocessing for worship music segments. The API returns timestamps in seconds (float); convert to SRT/VTT format for subtitles using simple formatting.

Message retention: 12 hours on free tier, configurable (7 days) when self-hosted. No built-in analytics or delivery confirmation—consider logging notifications to status database.

## Component 14: Status Tracking Database

**Recommended Project**: better-sqlite3  
- **URL**: https://github.com/WiseLibs/better-sqlite3  
- **Stars**: Widely used | **Last commit**: Active | **License**: MIT  
- **Language**: C++ bindings for Node.js

**Why this one**: Fastest SQLite library for Node.js with synchronous API (simpler than async), handling 2,000+ queries/second with proper indexing. Single-file database perfect for tracking episode status, upload history, approval decisions, and metadata without operational complexity of PostgreSQL/MySQL. While Node.js v22+ includes native `node:sqlite` module (zero dependencies), it's experimental (Stability 1.1) with potential breaking changes—better-sqlite3's battle-testing and 10+ years of production use provide stability. lowdb (22,500 stars) is simpler (JSON file) but performance degrades beyond 1,000 records and lacks SQL querying for dashboard reports.

**How to use**: Library import via npm

**Integration with n8n**: **Easy** - Install via npm, use n8n Code node to execute queries. Workflow: File detected → Insert episode record with "processing" status → Update status as workflow progresses (transcribing, pending approval, uploading, published) → Dashboard queries status table for UI. Use WAL mode (`PRAGMA journal_mode=WAL`) for concurrent read/write—multiple n8n workflows can query status while uploads progress.

**Code snippets to grab**: Database initialization with WAL mode from examples, prepared statements for parameterized queries (prevent SQL injection), transaction handling for atomic updates (upload status + metadata insert), full-text search setup for episode title/description queries, backup strategy (copy `.db` file while running). The synchronous API simplifies error handling compared to async libraries.

**Honorable mentions**:  
- **Node.js native SQLite** (v22+): Zero dependencies, built-in, but experimental status risky for production  
- **lowdb** (22,500 stars, MIT): Perfect for \u003c1,000 episodes; simpler than SQL but no complex queries

**Gotchas**: Synchronous API blocks Node.js event loop—for heavy queries (full table scans, complex JOINs), use worker threads or limit query complexity. SQLite handles up to ~1 million rows efficiently (sufficient for years of episodes), but concurrent writes are limited (use BullMQ to serialize writes if needed). **Compilation required**: better-sqlite3 includes C++ bindings requiring `node-gyp` at install time; prebuilt binaries available for common platforms (Linux, Mac, Windows), but ARM servers may need compilation (budget 5 minutes). Backup strategy: Use `cp database.db database.backup.db` on cron schedule; SQLite's single-file design makes backups trivial.

```python
from flask import Flask, request, jsonify
from faster_whisper import WhisperModel

app = Flask(__name__)
model = WhisperModel("large-v2", device="cuda", compute_type="int8")

@app.route('/transcribe', methods=['POST'])
def transcribe():
    video_path = request.json['video_path']
    segments, info = model.transcribe(video_path, batch_size=8)
    transcript = [{"start": s.start, "end": s.end, "text": s.text} for s in segments]

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000)
```

## Important Caveats

### License Compatibility
**The good news**: Most recommendations use MIT or Apache 2.0 (most permissive). **The exception**: Rumble's FelixEdenborgh uploader has unspecified license—assume permissive for personal use but verify before commercial distribution.

**License categories in your stack**:
- **MIT** (most permissive): Chokidar, BullMQ, better-sqlite3, Dropbox SDK, basic-ftp, faster-whisper, whisper.cpp, shadcn-dashboard, React-Admin, Claude SDK, Vercel AI SDK
- **Apache 2.0** (permissive, patent grant): youtubeuploader, ntfy (dual-licensed), Playwright, Vercel AI SDK, Signiant examples
- **GPL variants** (copyleft—avoid mixing): tokland/youtube-upload (GPLv3), fswatch (GPL-3.0, but Apache option available)
- **BSD variants** (permissive): Apprise (BSD-2-Clause), WhisperX (BSD-4-Clause)
- **Fair-code** (Apache + Commons Clause): n8n (can use freely, cannot resell as service)

**Mixing strategy**: You can combine MIT, Apache 2.0, BSD in one system without restrictions. Avoid GPL libraries (tokland/youtube-upload) in closed-source derivatives. n8n's Fair-code license allows ministry use but prohibits selling n8n-as-a-service competing with n8n.io.

### Tracking Upstream Changes When Forking

**For dashboard template** (Kiranism/next-shadcn-dashboard-starter):
- **Fork strategy**: Click "Fork" on GitHub, clone your fork, add upstream remote: `git remote add upstream https://github.com/Kiranism/next-shadcn-dashboard-starter.git`
- **Update cadence**: Check upstream monthly (`git fetch upstream`), review changes (`git log upstream/main`), merge security fixes only (`git merge upstream/main` selectively)
- **Customization isolation**: Keep your episode-specific code in separate feature folders (`app/episodes/`, `components/episodes/`), minimize changes to core components—makes upstream merges easier

**For libraries** (Chokidar, BullMQ, etc.):
- **Don't fork**: Use as npm dependencies, upgrade via `npm update` when needed
- **Version pinning**: Lock major versions in `package.json` (e.g., `"chokidar": "^4.0.0"` not `"chokidar": "*"`)—prevents breaking changes
- **Upgrade testing**: Test in development environment before production deployments

**For CLI tools** (youtubeuploader, basic-ftp):
- **Download releases**: Use GitHub Releases page for binaries, don't compile from source unless necessary
- **Version tracking**: Document versions in your deployment notes (e.g., "youtubeuploader v1.25.5"), subscribe to repo's Releases RSS feed

### Import as Library vs. Copy Code

**Always import as library** (don't copy source):
- BullMQ, Chokidar, better-sqlite3, Dropbox SDK, basic-ftp, Vercel AI SDK, Playwright
- **Why**: Active maintenance, security patches, bug fixes—copying source means you maintain forever
- **Exception**: If library is abandoned (\u003e2 years no commits, unanswered issues), consider forking (hasn't happened with any recommendations)

**Copy code patterns** (not whole projects):
- Approval workflow patterns from n8n templates—these are workflow configurations, not code
- Specific integration examples from SDK documentation (OAuth flows, chunked upload patterns)
- Configuration snippets (Chokidar file watcher options, BullMQ retry strategies)

**Run as-is** (no modification needed):
- youtubeuploader binary—pure CLI tool
- ntfy service—HTTP API, no code changes needed
- faster-whisper in Docker—wrapper is minimal glue code

### Red Flags for Abandoned Projects

**Do NOT use if project has ALL THREE**:
1. **Zero activity**: Last commit \u003e2 years ago (as of May 2026, avoid anything before 2024)
2. **Low adoption**: \u003c100 stars AND \u003c10 forks (indicates no community)
3. **Ignored issues**: 20+ open issues with no maintainer responses in 6+ months

**Examples of abandoned projects to avoid**:
- Old Selenium-based YouTube uploaders from 2018-2019
- Unmaintained Whisper wrappers from early 2023 (Whisper released Nov 2022, many experimental wrappers died)
- Ancient FTP libraries (node-ftp from 2015—use basic-ftp instead)

**Green flags** (all recommendations have these):
- Active commits in last 6 months
- Maintainer(s) responding to issues within 2 weeks
- Regular releases (even if just patch versions)
- 500+ stars OR official SDK/tool (Signiant, Dropbox, Google APIs)
- Production usage examples in README or Issues

**Gray area**: Low stars (\u003c100) but recent activity—could be new project gaining traction. Evaluate on merit: code quality, documentation, active maintainer. Example: FelixEdenborgh/Rumble-uploader fails this test (4 stars, abandoned 2023)—use for reference only, not production.

### Production Deployment Checklist

Before going live:
1. **Test failure scenarios**: Kill Redis mid-job (BullMQ should retry), corrupt video file (transcription should fail gracefully), API rate limits (uploads should queue and retry)
2. **Implement monitoring**: ntfy alerts for critical failures, dashboard showing stuck jobs (status unchanged \u003e2 hours), database queries for upload success rates
3. **Backup strategies**: SQLite database daily backups (cron job copying .db file), episode metadata JSON exports, OAuth tokens securely stored
4. **Rate limiting**: YouTube 6 videos/24 hours (stagger uploads), OpenAI 10K requests/min (throttle AI generation), Dropbox API limits (check docs)
5. **Cost tracking**: OpenAI API usage (\$0.50-2 per episode for metadata), managed Redis (~$20/month), Signiant Media Shuttle (enterprise pricing), cloud compute for transcription
6. **Security**: API keys in n8n credentials (not hardcoded), ntfy self-hosted for sensitive data, HTTPS for all webhook endpoints, HMAC signing for approval URLs
7. **Documentation**: Document each component's purpose, how to restart services, troubleshooting common errors (OAuth expired, Redis connection failed, disk full)
